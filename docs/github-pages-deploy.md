# GitHub + GitHub Pages: как обновлять сайт

Документ для людей и агентов: **один каноничный сценарий**, без GitHub Connector и без догадок.

## 1. Что где лежит

| Что | Где в GitHub | Зачем |
|-----|----------------|------|
| Исходники Astro (`src/`, `public/`, конфиги) | ветка **`main`** | разработка, история |
| Собранный статический сайт | ветка **`gh-pages`** | то, что отдаёт **GitHub Pages** |
| Папка `dist/` локально | **не в git** (в `.gitignore`) | появляется только после `npm run build` |

Опубликованный превью-URL (с учётом `base` в `astro.config.mjs`):

`https://timurgromov.github.io/timurgromov-site-2026_2/`

В репозитории на GitHub: **Settings → Pages → Build and deployment → Deploy from a branch → Branch: `gh-pages` / folder: `/ (root)`**.

## 2. Аутентификация: только обычный git по HTTPS

- Пуш делается **из терминала**: `git push`, не через встроенные «GitHub Connect» агентов без прав записи.
- `git remote -v` должен показывать `https://github.com/timurgromov/timurgromov-site-2026_2.git` (или актуальный owner/repo).
- Логин: **GitHub username**; пароль: **Personal Access Token (PAT)** с правами на репозиторий (классически `repo`). Креды часто лежат в **macOS Keychain** после первого успешного пуша.

Важно про PAT:

- Для пуша **обычных файлов** достаточно scope вроде **`repo`**.
- Отдельное правило GitHub: чтобы пушить файлы в **`.github/workflows/*.yml`**, у токена должен быть scope **`workflow`**. Если его нет — ветка `main` всё равно пушится, а workflow-файлы GitHub отклонит. В этом проекте депLOY на Pages завязан на ветку **`gh-pages`**, а не на Actions (если не меняли настройки).

## 3. Полный цикл после правок в коде

Рабочая директория — корень репозитория (где `package.json`).

### Шаг A — проверка сборки

```bash
ASTRO_TELEMETRY_DISABLED=1 npm run build
```

Должно завершиться без ошибок. Результат — каталог `dist/`.

### Шаг B — исходники на GitHub (`main`)

```bash
git status
git add -A
git commit -m "краткое описание изменений"
```

Если локальная ветка не называется `main`, но нужно обновить **`origin/main`**:

```bash
git push origin HEAD:main
```

Если ты уже на `main` и upstream настроен:

```bash
git push origin main
```

### Шаг C — опубликовать билд на GitHub Pages (`gh-pages`)

После успешного `npm run build` отправить содержимое `dist/` в ветку **`gh-pages`**. Практичный вариант без установки пакета в проект:

```bash
npx --yes gh-pages -d dist -b gh-pages
```

Флаги по смыслу:

- `-d dist` — что заливать;
- `-b gh-pages` — в какую ветку.

При первом запуске `npx` может скачать пакет `gh-pages`; для пуша снова нужны те же HTTPS-креды, что и для обычного `git push`.

Альтернатива (если `npx gh-pages` не устраивает): вручную скопировать файлы из `dist/` в локальный checkout ветки `gh-pages` и сделать `git commit` + `git push origin gh-pages`. Смысл тот же: **в `gh-pages` должны лежать файлы билда в корне ветки**.

### Шаг D — проверка live

1. Подождать 1–3 минуты (иногда дольше).
2. Открыть превью-URL (см. §1), сделать жёсткое обновление страницы (без кеша), при необходимости инкогнито.

Если в браузере старая версия — чаще всего кеш или ещё не подтянулась сборка Pages.

## 4. Частые ошибки (чтобы агенты не «тупили»)

1. **Запушили только `main`, забыли `gh-pages`** — на Pages старый билд. Нужен шаг C после каждого значимого изменения вёрстки/ассетов.
2. **Сломанные пути к картинкам на Pages** — в Astro заданы `site` и **`base`** в `astro.config.mjs`; ссылки на файлы из `public/` должны собираться с учётом `base` (в проекте для этого используется нормализованный base + helper `asset()` в `index.astro`). Не хардкодить `/images/...` без префикса репозитория.
3. **Путаница с Connector / API агента** — они не заменяют `git push` с машины пользователя, если нет прав записи. Рабочая схема: **локальный терминал + git + PAT в Keychain**.
4. **`Could not resolve host: github.com`** — сеть/DNS на машине; push не дошёл, GitHub не обновился.

## 5. Опциональная ветка `gh-pages-source`

Если в remote есть ветка **`gh-pages-source`**, это отдельный согласованный процесс (история/бэкап исходников под деплой). Текущий канон для **публичного сайта** — ветка **`gh-pages`** с содержимым `dist/`. Не путать с `main`.

## 6. Краткая шпаргалка одной строкой

```text
npm run build && git push origin HEAD:main && npx --yes gh-pages -d dist -b gh-pages
```

(Коммит в `main` перед этим сделать отдельно, если есть незакоммиченные изменения.)
