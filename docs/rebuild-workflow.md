# Rebuild Workflow

## 1. Purpose
Этот документ — единый рабочий план восстановления сайта `timurgromov.ru`.

Он нужен, чтобы:
- не было нескольких конфликтующих планов;
- работа шла в одном порядке;
- каждая секция восстанавливалась последовательно;
- проект не превращался в хаотичный набор правок.

## 2. Source Of Truth
Всегда использовать источники в таком порядке:
1. `page62008353.html`
2. `files/page62008353body.html`
3. живой сайт `https://timurgromov.ru/`
4. экспортированные папки `images/`, `css/`, `js/` и другие файлы export
5. пользовательские скриншоты
6. служебная документация проекта

Правило:
- export — главный источник структуры и контента;
- живой сайт — главный визуальный ориентир;
- если есть сомнение, сначала проверять реальные файлы, а не предполагать.

## 3. Hard Rules
- Не делать новый дизайн.
- Не менять порядок секций без подтверждения export.
- Не придумывать новые блоки.
- Не придумывать тексты, CTA, цены, отзывы, кейсы, FAQ.
- Не улучшать UX «по своему вкусу».
- Не начинать с нижних секций.
- Не трогать видео до завершения статической структуры.
- Не подключать формы раньше времени.
- Не плодить новые планы в других документах.

## 4. Work Order
Восстановление идет строго сверху вниз.

Порядок:
1. глобальная база: шрифты, цвета, spacing, фон, базовые контейнеры, кнопки
2. fixed header
3. hero
4. about
5. «Как я веду»
6. «Полезности»
7. price
8. cases
9. contacts
10. popup/menu overlay
11. video
12. forms

Правило: нельзя переходить ниже, пока верхняя секция не приведена близко к оригиналу.

## 5. Section Rule
Каждая секция проходит один и тот же цикл:
1. найти секцию в export;
2. найти соответствующий блок на живом сайте;
3. восстановить только эту секцию;
4. сверить результат;
5. только потом переходить к следующей.

Нельзя: править сразу много несвязанных блоков; смешивать работу по hero, cases и popup одновременно; накапливать «временные» решения.

## 6. Verification Rule
После каждой завершенной секции обязательно:
1. сверка с export;
2. сверка с `timurgromov.ru`;
3. локальная сборка;
4. визуальная проверка;
5. только после этого переход дальше.

## 7. Change Cycle
Для каждого значимого изменения:
1. прочитать реальные файлы;
2. сделать минимальную правку;
3. выполнить `build`;
4. при наличии линтера выполнить `lint`;
5. commit;
6. push;
7. deploy на GitHub Pages;
8. проверить live URL.

## 8. Deploy Rule
Deploy должен быть простым и повторяемым: локальная проверка → commit → push → GitHub Pages → проверка опубликованной версии.

**Конкретные команды и нюансы (main vs gh-pages, HTTPS, PAT, `npx gh-pages`):** см. `docs/github-pages-deploy.md`.

## 9. Popup Rule
Popup и burger/menu overlay — состояния интерфейса внутри одного лендинга. Не делать popup отдельными страницами; не ломать якоря и popup hooks из оригинала.

## 10. Video Rule
Видео — только после завершения статического восстановления. Порядок: структура → popup-state → видео → формы.

## 11. Forms Rule
Формы — после статического rebuild, popup/menu и стабилизации deploy.

## 12. Definition Of Done
Этап завершён, когда структура и порядок совпадают с оригиналом, тексты и картинки из export, popup/menu работают, сверка с live и GitHub Pages пройдена.

## 13. Agent Startup Rule
Каждый новый чат или агент:
1. `docs/project-spec.md`
2. `docs/rebuild-workflow.md`
3. `docs/source-of-truth.md`
4. `docs/github-pages-deploy.md` (перед любым deploy или если «не обновился сайт»)
5. `docs/section-map.md` (при необходимости)
6. export-файлы
7. только потом правки в коде.

## 14. Summary
Один план, один порядок, одна секция за раз, проверка после каждой секции, без самодеятельности.
