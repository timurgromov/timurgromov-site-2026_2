# Source Of Truth

## 1. Purpose
Этот файл фиксирует, каким источникам доверять при восстановлении сайта `timurgromov.ru`.

Нужен для того, чтобы:
- не было споров, откуда брать правильную структуру;
- не появлялись выдуманные блоки;
- любой новый чат или новый агент работал по одной и той же базе.

## 2. Priority Order
Всегда использовать источники в таком порядке:

1. `page62008353.html`
2. `files/page62008353body.html`
3. живой сайт `https://timurgromov.ru/`
4. экспортированные ассеты:
   - `images/`
   - `css/`
   - `js/`
   - другие файлы export
5. пользовательские скриншоты оригинала
6. документация проекта:
   - `docs/project-spec.md`
   - `docs/rebuild-workflow.md`
   - `docs/section-map.md`

## 3. Interpretation Rule
- Tilda-export — главный источник структуры, контента, popup hooks и порядка блоков.
- Живой сайт — главный визуальный ориентир.
- Скриншоты нужны для уточнения спорных деталей.
- Локальная документация не должна противоречить export.

Если есть расхождение:
- по структуре доверять export;
- по визуальным нюансам сверять export + live site + screenshots;
- не принимать решения “на глаз”, если можно проверить файл.

## 4. What Is Allowed
Можно:
- восстанавливать блоки по export;
- переиспользовать тексты, изображения и popup hooks;
- повторять структуру и порядок секций;
- адаптировать кодовую реализацию, если визуальный и структурный результат остается тем же.

## 5. What Is Not Allowed
Нельзя:
- придумывать новые секции;
- добавлять новые тексты;
- менять CTA без подтверждения источниками;
- менять цены, отзывы, FAQ, кейсы без подтверждения export;
- менять порядок блоков по собственному усмотрению;
- делать редизайн;
- заменять точное восстановление на “похожую современную версию”.

## 6. Visual Verification Rule
После каждой значимой секции нужно сверять:
1. export
2. live original
3. опубликованную версию проекта

Не проверять только локально.
Не сверять только по памяти.

## 7. Live Verification Rule
Для этого проекта опубликованный `GitHub Pages` URL является обязательной точкой проверки после deploy.

Локальная сборка нужна для скорости, но не считается финальным подтверждением, если изменены:
- секции страницы
- popup
- ассеты
- маршруты
- стили
- якоря
- интерактивные состояния

## 8. Operational Rule
Каждый новый чат или агент должен начинать с чтения:
1. `docs/project-spec.md`
2. `docs/rebuild-workflow.md`
3. `docs/source-of-truth.md`
4. `docs/section-map.md`
5. export-файлы

Только после этого можно вносить изменения.

## 9. Final Principle
Если есть сомнение:
- остановиться;
- открыть реальные файлы;
- сверить с оригиналом;
- не делать предположения.
